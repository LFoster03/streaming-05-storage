## Custom Project

### Dataset

The dataset used is `data/weather_readings.csv`, a simulated stream
of weather station readings from 5 US cities.

Each record represents one sensor reading from a weather station and
contains the following fields:

- `reading_id`: unique identifier for the reading event
- `datetime`: ISO 8601 timestamp of when the reading was taken
- `station_id`: which station sent the reading (e.g. STN-MIA)
- `unit_system`: measurement system used (imperial)
- `temperature`: air temperature in Fahrenheit
- `humidity_pct`: relative humidity as a percentage
- `wind_speed`: wind speed in miles per hour
- `condition`: weather condition (sunny, cloudy, rainy, windy, humid)
- `device_id`: which sensor device sent the reading

This is a custom dataset, not the original sales dataset. Two
reference tables support validation:

- `data/stations.csv`: station location details for 5 US cities
- `data/units.csv`: unit system labels (imperial/metric)

### Kafka Messages

The producer reads `weather_readings.csv` and sends one message per
row to the Kafka topic `streaming-05-weather-foster`.

Each message is validated before sending:

- Required fields are checked against the weather data contract
- `station_id` is checked against `stations.csv`
- `unit_system` is checked against `units.csv`
- `condition` is checked against a set of allowed values

The message key is `station_id`, which ensures all readings from the
same station are routed to the same Kafka partition and stay in order.

Invalid messages are rejected and written to
`data/output/producer_rejected_weather.csv` instead of being sent.

### Consumer Processing

The consumer subscribes to `streaming-05-weather-foster` and reads
all available messages from the beginning of the topic.

For each message received, the consumer:

1. Validates required fields and allowed values
2. Computes `heat_index` using the Rothfusz regression formula
   from the National Weather Service (activates at >= 80F, >= 40%
   humidity)
3. Computes `wind_chill` using the National Weather Service formula
   (activates at <= 50F and wind speed > 3 mph)
4. Writes the enriched record to `data/output/weather.duckdb`
5. Appends the enriched record to `data/output/consumed_weather.csv`
6. Logs running temperature statistics after each message

The consumer stops when all messages are consumed or when no new
message arrives within the timeout window (default 10 seconds).

The output CSV contains all original fields plus:
`heat_index`, `wind_chill`, `_kafka_key`, `_kafka_partition`,
and `_kafka_offset`.

### Results

The producer successfully sent all 10 weather readings to the topic
with no rejections. The consumer consumed all 10 messages and wrote
them to both DuckDB and the output CSV.

Key observations from the output:

- Miami (STN-MIA) at 84.6F and 91% humidity produced a heat index
  of 100.6F, a 16 degree feels-like difference due to humidity.
- Dallas (STN-DFW) at 72.4F did not trigger either formula, so
  heat_index and wind_chill both equal the raw temperature.
- Seattle (STN-SEA) at 54.1F was just above the wind chill
  threshold and did not trigger the formula.
- The DuckDB summary logged average temperature, heat index, and
  wind chill per station after all messages were consumed.

### Interpretation

The weather pipeline demonstrated the same Kafka streaming workflow
as the sales example but with a completely different domain.

What changed from the original example:

- The event stream represents sensor readings instead of purchases.
- Derived fields use meteorological formulas instead of tax math.
- The partition key is a location identifier (station) rather than
  a sales region.
- The consumer tracks temperature statistics instead of revenue.

What I learned from watching messages move through Kafka:

- Each message moves through the pipeline independently, so derived
  fields are computed one reading at a time as they arrive.
- The partition key ensures station readings stay in chronological
  order, which matters for time-series weather data.
- Separating raw measurements from derived calculations keeps the
  producer simple and puts business logic in the consumer.

What the stream could tell a business or organization:

- A utility company could monitor real-time heat index to anticipate
  spikes in energy demand during hot humid weather.
- An outdoor events company could watch wind chill to make
  decisions about event cancellations.
- A logistics company could route deliveries based on live
  conditions across regional stations.

Business intelligence gained:

- High humidity locations like Miami produce dangerous heat index
  values even at moderate temperatures, which raw temperature
  alone would not reveal.
- Streaming derived fields in real time means decisions can be made
  as conditions change, not after the fact.
